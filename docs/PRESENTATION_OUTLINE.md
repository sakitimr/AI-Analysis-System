### 第一幕：问题引入（30 秒）

**幻灯片**：竞品分析的传统痛点

> "竞品分析是企业产品团队的常规工作，涉及信息搜集、功能对比、用户评价整理、SWOT 分析、报告撰写等多个环节。传统方式依赖分析师手动完成，一个竞品就要花 2-4 小时。"

**过渡**：
> "我的方案是用 4 个专职 AI Agent 组成一个数字调研小组，自动完成全链路。"

### 第二幕：系统架构（60 秒）

**幻灯片**：四层架构图

> "系统采用四层架构：前端交互层用 Streamlit，服务编排层用 LangGraph StateGraph，知识存储层用 Pydantic Schemas + SQLite，基础设施层提供配置、日志、异常处理。"

**幻灯片**：DAG 任务流转图

> "核心编排是一个 DAG：Collector 采集 → Analyst 分析 → Writer 撰写 → QC 质检。关键设计是条件路由——QC 不通过时，按问题类型精准打回对应 Agent，最多 3 轮迭代。这是真正的反馈闭环，不是伪闭环。"

**关键陈述**：
> "这里有一个重要设计决策：为什么选 LangGraph 而不是 CrewAI？因为课题明确要求 Agent 间采用结构化消息传递，而非自然语言对话。LangGraph 的 StateGraph 通过 TypedDict 传递结构化数据，天然支持条件路由和反馈闭环。CrewAI 偏自然语言对话，不符合课题要求。"

### 第三幕：现场演示（120 秒）

**操作 1：CLI 运行（30 秒）**

```bash

```

**解说**：
> "我现在运行 CLI 模式。可以看到它加载了预采集的样本数据——这是离线模式，确保即使 API 不可用也能完整演示。"

**操作 2：观察执行（30 秒）**

> "现在看到 Collector 正在执行——它有两个工作模式：对于已知维度如功能对比和定价，使用硬编码的搜索模板；对于未知维度，通过 LLM 自适应生成搜索查询。这是前瞻性设计的一个体现。"

> "Analyst 正在构建功能对比矩阵。它使用 LLM 数据驱动的 0-3 评分，对每个功能维度独立评分。如果 LLM 评分失败，会自动降级到 heuristic 评分——基于数据详度的容错机制。"

**操作 3：QC 闭环（30 秒）**

**操作 5：Web UI（可选，30 秒）**
```bash

```

> "如果时间允许，我可以展示 Streamlit Web UI——任务创建、DAG 实时监控、报告查看和决策回放。"

### 第四幕：技术亮点（60 秒）

**幻灯片**：四大技术亮点

1. **真实反馈闭环**
   > "QC 不是简单地输出 pass/fail，而是产出结构化的 QCVerdict，每个 issue 含 target_agent 字段。Router 按优先级精准打回——collector 问题带 collection_hints 重采，analyst 问题只重新分析，writer 问题只重新撰写。3 轮上限后降级输出，确保不陷入死循环。实测验证了 3 轮迭代中得分从 0.40 提升到 0.65+。"

2. **结构化通信**
   > "Agent 之间不传自然语言。Collector 产出 CollectedData，Analyst 读取后产出 AnalysisResult，Writer 再读取后产出 FinalReport。每个阶段都是强类型 Pydantic Model。这完全符合课题要求的 function calling 范式。"

3. **三层幻觉抑制**
   > "第一层 Prompt 约束——Collector 强制要求 ONLY use from search results；第二层规则校验——正则检查 source_url 是否都在采集数据集中；第三层 LLM 自检——QC 抽样检查结论与原文一致性。"

4. **工程完整度**
   > "端到端全链路：CLI + API + Web UI + 单元测试 15/15。可观测性：每个 Agent 的 Prompt/输入/输出/Token/耗时全记录。稳定性：tenacity 指数退避重试 + 离线降级 + 3 轮迭代上限。"

### 第五幕：总结与展望（30 秒）

> "总结三点核心价值：第一，效率提升——传统 2-4 小时的分析压缩到 8 分钟；第二，质量保障——结构化 Schema + 质检闭环确保输出一致性；第三，工程完整——从采集到报告到前端的端到端系统。"

> "展望方向：支持更多信息源如社交媒体和学术论文，动态 Schema 演化适应不同行业，以及 Agent 自评估机制持续优化。"

## 三、预期问题与应答

### Q1: 为什么选 LangGraph 而不是 CrewAI？

**A:** 课题明确要求 Agent 间采用结构化消息传递（类似 function calling），非纯自然语言对话。LangGraph 的 StateGraph 通过 TypedDict 传递结构化数据，条件路由天然支持反馈闭环。CrewAI 偏自然语言对话，不符合要求。

### Q2: 反馈闭环是真实的还是模拟的？

**A:** 真实可触发的反馈闭环。QC 产出 QCVerdict，每个 issue 指明 target_agent。Router 根据 issue 类型精准路由。Collector 被打回时使用 collection_hints 仅重采缺失维度。实测 3 轮迭代中输出质量逐轮改善。

### Q3: 如何确保 Agent 输出的结构化一致性？

**A:** 三层保障：(1) Pydantic 强类型定义，运行时校验；(2) safe_invoke_json 自动解析+修复，失败则让 LLM 自修复 JSON；(3) 降级兜底——仍失败则用默认值填充，标记为 incomplete。

### Q4: 幻觉问题怎么处理？

**A:** 三层防护：(1) Collector Prompt 强制"仅从搜索结果提取"；(2) Pydantic 字段约束（source_url 必填）；(3) QC 的 _llm_check 抽样验证—"请验证以下结论是否与原文一致"。

### Q5: 如何处理 API 不可用的情况？

**A:** 离线模式。sample_data.py 预采集了三竞品的完整数据。`--online` 关闭时直接使用本地数据，全链路完整运行。确保答辩现场即使网络出问题也能完整演示。

### Q6: 这个系统的创新点在哪？

**A:** (1) 自适应查询生成——未知维度通过 LLM 动态生成搜索查询；(2) 双轨评分策略——LLM 评分 + heuristic 降级自动切换；(3) 精准重采机制——QC 打回只重采缺失维度，非全量重跑；(4) 三层安全 JSON 解析——解析失败让 LLM 自我修复。

### Q7: 架构设计的自主性如何体现？

**A:** 虽然是 AI 辅助开发（TRAE IDE + 贾维斯协助代码实现），但以下关键决策由我自主完成：(1) 四层架构设计；(2) LangGraph vs CrewAI 的技术选型分析；(3) Pydantic Schema 的字段设计；(4) 反馈闭环的路由策略；(5) 三层幻觉抑制方案；(6) 双轨评分降级机制。AI 工具的角色是"高效的代码执行者"而非"架构决策者"。

### Q8: 系统如何扩展支持其他行业的竞品分析？

**A:** 通过三个层面：(1) `--competitors` 和 `--dimensions` 参数支持任意竞品和维度；(2) 自适应查询生成——遇到未知维度时 LLM 自动生成搜索查询；(3) Schema 设计为通用竞品分析结构（功能树/定价模型/用户画像），适用于大多数 B2B SaaS 产品。
